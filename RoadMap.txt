     Roadmap: BTC Bloom Scanner Pro
Дорожная карта развития проекта, ориентированная на масштабирование, производительность и создание распределённой вычислительной экосистемы.
     Видение проекта
Превратить сканер из локального инструмента в высокопроизводительную платформу для криптографического поиска и восстановления, способную работать на одном ПК, в локальной сети или в облачном кластере с гибридной загрузкой CPU+GPU.
     Фаза 1: Стабилизация и оптимизация (v1.2 – Текущая)
Фокус: Доведение текущей Python-архитектуры до идеала, улучшение UX и надёжности.
Задача
     Оптимизация памяти- Использование memoryview и mmap для загрузки Bloom-фильтров без полного копирования в RAM
     Расширенная аналитика- Графики распределения Hits, тепловая карта скорости, экспорт метрик в CSV
     Улучшенная безопасность- Шифрование found_wallets.txt, автоскрытие сид-фраз в GUI после 30 сек
     PWA-режим- Установка веб-интерфейса как десктопного приложения, офлайн-кэш логов
     Тестовый режим- Виртуальный API-эмулятор для отладки Recovery без реальных запросов


     Фаза 2: BloomNet – Распределённые вычисления (v2.0)
Фокус: Создание кластерной системы, аналогичной SwarmAgent, но специализированной под крипто-сканирование.
     Архитектура BloomNet
[Master Node] ←→ [Redis/RabbitMQ] ←→ [Worker 1] [Worker 2] ... [Worker N]
      ↓                   ↓                ↓          ↓
   Dashboard         Task Queue        CPU/GPU      CPU/GPU
     Ключевые возможности:
Мастер-воркер модель: Центральный узел распределяет диапазоны адресов или сид-пространства между узлами.
Автообнаружение: mDNS/Bonjour для LAN, токены + TLS для WAN.
Динамический балансировщик: Автоматическое перераспределение задач при отключении воркеров.
Безопасный канал: gRPC over TLS, аутентификация по сертификатам, шифрование результатов.
Централизованный дашборд: Веб-интерфейс мастера с картой узлов, загрузкой, статистикой Hits в реальном времени.
Аналогия со SwarmAgent: Децентрализованная координация, но с централизованным контролем задач и агрегацией результатов.
     Технологический стек:
gRPC / ZeroMQ для коммуникации
Redis для очередей и состояния
Docker + docker-compose для развёртывания воркеров
Prometheus + Grafana для мониторинга кластера
     Фаза 3: BloomCore – Переход на C/C++ (v3.0)
Фокус: Переписывание вычислительного ядра на C/C++ с сохранением Python-обёртки для GUI и оркестрации.
     Стратегия миграции:
Слой                            Язык                            Роль
GUI / Web / API                 Python (Flask + SocketIO)       Оркестрация, интерфейс, сетевое взаимодействие
Bloom Engine                    C/C++ + pybind11                Битовые операции, каскадная проверка, mmap-загрузка
Crypto Core                     C (OpenSSL / libsodium)         SHA256, RIPEMD160, BIP39/BIP44 деривация
Scheduler                       Python → C (pthread/async)      Управление потоками, lock-free очереди
    Ожидаемый прирост:
Скорость генерации: ×10–25 (устранение GIL, прямая работа с памятью)
Потребление RAM: -40% (ручное управление буферами, отсутствие overhead Python-объектов)
Запуск фильтра: -70% времени (memory-mapped loading вместо bytearray(f.read()))
🛠 Инструменты:
pybind11 / Cython для биндингов
CMake для сборки кроссплатформенных библиотек
Valgrind / AddressSanitizer для отладки памяти
     Фаза 4: CUDA & Гибридный режим CPU+GPU (v3.5–v4.0)
Фокус: Распараллеливание вычислений на GPU с умным распределением нагрузки между CPU и GPU.
     Архитектура Hybrid Scheduler:
[Python Orchestrator]
       ↓
[CPU Thread Pool] ←→ I/O, API, GUI, Bloom Load, Recovery Logic
       ↓
[CUDA Manager] ←→ Kernel Launch, VRAM Management, PCIe Sync
       ↓
[GPU Kernels] → Address Gen → SHA256/RIPEMD160 → Bloom Check → Hit Buffer
     Возможности:
Автодетект: Поиск CUDA-совместимых GPU, fallback на CPU при отсутствии.
Гибридный режим: CPU обрабатывает сетевые задачи и UI, GPU – массовую генерацию и фильтрацию.
Динамический баланс: Автоматическая подстройка ratio CPU/GPU на основе температуры, загрузки VRAM и PCIe-пропускной способности.
Оптимизированные ядра:
cuRAND для криптографически стойкой генерации энтропии
Кастомные CUDA-ядра для SHA256+RIPEMD160 (агрегированные вызовы)
Побитовая проверка Bloom в shared memory
VRAM Management: Потоковая загрузка чанков фильтра, использование cudaHostAlloc для zero-copy transfers.
     Технологический стек:
CUDA C/C++ + nvcc
Numba / CuPy для прототипирования ядер
Thrust для сортировки/агрегации результатов
NVIDIA Nsight для профилирования
    Фаза 5: Экосистема и Enterprise (v5.0+)
Модуль                  Описание
Plugin System           Подключаемые модули для новых криптовалют, кастомных путей деривации, внешних API
Cloud Deploy            Готовые шаблоны для AWS/GCP/Azure, авто-масштабирование воркеров
Hardware Wallet Sync    Read-only проверка адресов Ledger/Trezor без экспорта ключей

Audit & Compliance      Логи действий, водяные знаки в экспортах, режим "только аудит

     Стратегия внедрения
Не ломать текущее: Все изменения внедряются обратно совместимо. Python-версия остаётся рабочей до полного завершения BloomCore.
Модульность: Ядро выносится в отдельную библиотеку libbloomcore. GUI общается с ней через стабильный C-API.
Тестирование: Для каждого этапа – набор unit/integration тестов на известных тестовых векторах (BIP39, BIP44, известные адреса).
Документация: Пошаговые гайды по развёртыванию BloomNet, компиляции CUDA-ядер, миграции на BloomCore.
     Примечания
    Python останется языком оркестрации и интерфейса даже после перехода на C/C++ и CUDA.
     Безопасность распределённых узлов: обязательное TLS, ротация ключей, аудит воркеров.
     CUDA-режим требует NVIDIA GPU с Compute Capability 6.0+. Поддержка AMD (ROCm) планируется в v4.5.
     BloomNet может работать как в закрытой LAN, так и через интернет с централизованным мастер-узлом.
     Roadmap is alive. Каждый этап будет сопровождаться релизными заметками, бенчмарками и миграционными гайдами.

Промежуточная Фаза Multi-Chain Expansion (v2.5 – v4.0)
Фокус: Расширение системы фильтрации и сканирования на другие блокчейн-сети с сохранением каскадной архитектуры и модульности.
    Поддерживаемые сети (план)

Сеть                Адреса                                  Стандарт деривации                      Хеш-функции                          Примечание
₿ Bitcoin (BTC)     Legacy (1...), SegWit (3..., bc1...)    BIP44 / BIP84                           SHA256 + RIPEMD160                   Уже реализовано
Ethereum (ETH)      0x... (40 hex)                          BIP44 m/44'/60'/0'/0                    Keccak-256                           EVM-совместимые
Litecoin (LTC)      Legacy (L...), SegWit (ltc1...)         BIP44                                   SHA256 + Scrypt/RIPEMD               Fork Bitcoin Core
Dogecoin (DOGE)     D..., A...                              BIP44 m/44'/3'/0'/0                     Scrypt + SHA256                      Низкая сложность
BCH Bitcoin Cash    bitcoincash:...                         BIP44 + CashAddr                        SHA256 + RIPEMD160                   Fork с изменённым форматом
TRX Tron            T... (Base58Check)                      BIP44 m/44'/195'/0'/0                   SHA256 + RIPEMD160                   EVM-совместимый

    Архитектура Multi-Chain Bloom System
    [Chain Registry]
       ↓
┌─────────────────────────────────┐
│ ChainConfig (per network)       │
│ • address_format (regex/prefix) │
│ • derivation_path (BIP44 index) │
│ • hash_functions (SHA/Keccak)   │
│ • bloom_params (error_rate)     │
│ • api_endpoints (balance check) │
└─────────────────────────────────┘
       ↓
[Multi-Chain Bloom Manager]
       ↓
┌─────────────┬─────────────┬─────────────┐
│ BTC Filters │ ETH Filters │ LTC Filters │
│ • fast      │ • fast      │ • fast      │
│ • strict    │ • strict    │ • strict    │
└─────────────┴─────────────┴─────────────┘
       ↓
[Unified Checker]
       ↓
• Input: mnemonic / pubkey / address
• Output: {chain: "BTC", hit: true, balance: 0.001}

    Ключевые возможности мультичейн-режима
    Универсальный генератор адресов
    Каскадные фильтры на каждую сеть
    Каждый блокчейн имеет пару фильтров: fast + strict
    Файлы именуются единообразно:
btc_addresses.bloom / btc_addresses_strict.bloom
eth_addresses.bloom / eth_addresses_strict.bloom
    Загрузка по требованию: фильтры грузятся только для выбранных в настройках сетей
     Единый API для проверки баланса
Recovery для разных сетей
Мнемоника (BIP39) общая, но пути деривации разные
Режим Recovery позволяет выбрать:
    Scan chains: [x] BTC [x] ETH [ ] LTC ...
Найденная фраза проверяется сразу по всем выбранным сетям
     BloomNet с поддержкой мультичейн
Воркеры могут специализироваться:
Worker #1 → BTC only | Worker #2 → ETH+ERC20 | Worker #3 → All chains
Мастер-узел агрегирует результаты и дедуплицирует находки
Балансировка: если по одной сети много хитов — автоматически масштабируем воркеры
